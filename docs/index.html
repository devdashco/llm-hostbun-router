llm.hostbun.cc

One OpenAI-compatible endpoint at https://llm.hostbun.cc/v1. Three providers, picked by model name: local local → on-prem GPU (free), wrappy claude* → real Claude via claudebox (claude.hostbun.cc), crazyrouter anything else → cloud relay (250+ models). Plus on-prem image generation (model: imagegen, SDXL on the pbox GPU). No API key needed; auth is handled server-side. Docs also at docs.llm.hostbun.cc.

Quick start — one URL, pick by model

One base URL https://llm.hostbun.cc/v1 for everything. The model name picks the provider: local → the local GPU (default model); claude* → wrappy / real Claude (claude.hostbun.cc); any other id → crazyrouter. Same OpenAI API either way.

Wrappy lane — claudebox model: claude*

Any model whose id starts with claude is routed to claude.hostbun.cc — a claudebox (Claude Code) wrapper backed by real Claude. The gateway injects the wrapper's token, so you don't send a key. Available: claude-opus-4-8, claude-opus-4-7, claude-opus-4-6, claude-sonnet-4-6, claude-opus-4-5. Supports chat completions + streaming via the OpenAI API. (The Anthropic-native /v1/messages shape on the wrapper is currently not working — use /v1/chat/completions.)

Automatic failover: claudebox is backed by a Claude subscription (no API key), so it has a weekly usage limit. When it's exhausted (or otherwise erroring) the gateway transparently re-routes the same request to the crazyrouter lane with the equivalent model — you keep getting answers instead of a "You've hit your weekly limit" message leaking back as the reply. This includes the case where the limit notice comes back as a normal 200 body. Disable with env WRAPPY_FALLBACK=0.

Because claudebox runs Claude Code (an agent identity scoped to software-engineering), non-coding tasks — especially ones that fetch a URL or judge a real person — may be refused. To make it behave like a plain model, pass the claudebox extra fields (forwarded untouched): "max_turns":1 and "disallowed_tools":["Bash","Read","Write","Edit","WebFetch"].

curl https://llm.hostbun.cc/v1/chat/completions -H "Content-Type: application/json" \
  -d '{"model":"claude-opus-4-8","messages":[{"role":"user","content":"hi"}]}'

Crazyrouter lane — cloud relay /v1

OpenAI-compatible aggregator (upstream api.crazyrouter.com/v1). Every request whose model isn't a local id (local / gemma / obliterated) and isn't a claude* model is proxied here with the key injected. Capabilities: chat, streaming ("stream":true), tool/function calling, structured outputs, vision, image/video/audio generation, embeddings, rerank. Full upstream reference: crazyrouter.com/docs.

# streaming
curl -N https://llm.hostbun.cc/v1/chat/completions -H "Content-Type: application/json" \
  -d '{"model":"gemini-2.5-pro","stream":true,"messages":[{"role":"user","content":"count to 5"}]}'

Image generation model: imagegen

Text-to-image runs on the pbox RTX 4090 (SDXL 1.0 + SDXL-Lightning, ~1.5 s for a 1024×1024 image), proxied to sdturbo.bofrid.dev with the upstream token injected. OpenAI-compatible /v1/images/generations — returns {"data":[{"b64_json":…}]} (we don't host URLs, so always base64). Use model: "imagegen" ("sd-turbo" still works as a legacy alias).

FieldMeaning
promptText prompt. Optional if template is given.
templateNamed prompt template (see GET /v1/templates): profile-woman-back, profile-man-back — vacation profile pics that randomize hair / outfit / scene / lighting per image.
varsPin specific template slots, e.g. {"hair":"jet black"}; the rest stay random.
loraA registered LoRA name (see GET /v1/loras), an HF repo id, or a local path. Stacked on the speed LoRA; the LoRA's trigger phrase is auto-added to the prompt.
lora_scaleOverride the LoRA strength (default per-LoRA).
n · size · seed · stepsCount (1–8), WxH (default 1024x1024), seed, steps (default 8).

Templates randomize per image and return what they picked in data[i].template_vars + data[i].revised_prompt. Named LoRAs currently: watercolor (soft watercolor — great for article header images), nostalgic-cinema (warm film look), hair-bun (top-knot hairstyle). Add more by registering them in the service.

# plain prompt
curl https://llm.hostbun.cc/v1/images/generations -H "Content-Type: application/json" \
  -d '{"model":"imagegen","prompt":"a small red cabin by a Swedish lake","n":1}'

# article header image — watercolor LoRA by name
curl https://llm.hostbun.cc/v1/images/generations -H "Content-Type: application/json" \
  -d '{"model":"imagegen","lora":"watercolor","prompt":"a cozy reading nook by a window"}'

# template with randomized hair/scene, 4 variations
curl https://llm.hostbun.cc/v1/images/generations -H "Content-Type: application/json" \
  -d '{"model":"imagegen","template":"profile-woman-back","n":4}'

# discover templates + LoRAs
curl https://llm.hostbun.cc/v1/templates
curl https://llm.hostbun.cc/v1/loras
# save the first image (b64) to a file
curl -s https://llm.hostbun.cc/v1/images/generations -H "Content-Type: application/json" \
  -d '{"model":"imagegen","lora":"watercolor","prompt":"row of colorful Swedish houses"}' \
  | python3 -c "import sys,json,base64;open('out.png','wb').write(base64.b64decode(json.load(sys.stdin)['data'][0]['b64_json']))"

Structured / JSON output enforced

When you send response_format: {"type":"json_object"} (or {"type":"json_schema",…}) on /v1/chat/completions, the gateway verifies the model actually returned valid JSON before handing the reply back — it doesn't trust the model to honour the flag. This works the same on every lane (local, wrappy, crazyrouter), including models that don't natively support structured outputs.

  • If the reply parses as JSON → returned unchanged.
  • If it's wrapped in a ```json … ``` fence → the fence is stripped (auto-repaired), no extra round-trip.
  • If it still isn't valid JSON → the gateway re-prompts the same model with the parse error ("your reply was not valid JSON, remake it") and retries (default 2 retries).
  • If the model never complies → you get HTTP 422 with {"error":{"code":"json_validation_failed",…}} and the last raw content, instead of a malformed body that breaks your JSON.parse.

This even works for backends that don't natively support response_format. The wrappy wrapper returns an empty {} if you pass it response_format, and LM Studio (local) rejects json_object. For those the gateway strips response_format and instructs the model in-prompt instead (including your JSON Schema, when given), then validates the reply the same way — so model:"claude-…" with json_schema returns exactly the shape you asked for. The crazyrouter lane and local json_schema keep native constrained decoding.

Note: because the whole reply must be buffered to validate it, stream:true + a JSON response_format returns the content in a single reconstructed SSE chunk rather than token-by-token. Validation is structural JSON (not full JSON-Schema). Tune with env JSON_MAX_RETRIES (default 2); disable entirely with JSON_ENFORCE=0.

curl https://llm.hostbun.cc/v1/chat/completions -H "Content-Type: application/json" \
  -d '{"model":"local","response_format":{"type":"json_object"},
       "messages":[{"role":"user","content":"List 3 primary colors as a JSON array under key colors."}]}'
# → guaranteed-parseable JSON, or HTTP 422 json_validation_failed if the model refuses

Project attribution X-Project

Every calling app should identify itself so usage can be attributed per project (which app spent which tokens, which app is erroring). Declare it with the X-Project request header:

curl https://llm.hostbun.cc/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "X-Project: bofrid" \
  -d '{"model":"claude-sonnet-4-6","messages":[{"role":"user","content":"hi"}]}'

The gateway reads the project from the first of these it finds (normalised to a lowercase slug, max 64 chars):

  • X-Project header (preferred), or X-Project-Id
  • a top-level "project" field in the JSON body
  • "metadata":{"project":"…"} in the body
  • the OpenAI "user" field (fallback)

It's recorded on every call and surfaces in the admin call log + usage stats (filter GET /admin/api/calls?project=bofrid, breakdown in GET /admin/api/stats → byProject).

Enforcement (optional): when the gateway runs with REQUIRE_PROJECT=1 (or the admin toggle), any inference call that doesn't declare a project is rejected:

HTTP 400
{"error":{"message":"missing project attribution: send an 'X-Project' header (or a 'project' body field) identifying the calling app.",
          "type":"invalid_request_error","code":"project_required"}}

Default is off — the project is still recorded whenever you send it, just not mandatory.

Reasoning effort & extra params passthrough

The gateway only rewrites the model field (and, for JSON enforcement, response_format). Every other field is forwarded upstream untouched — so reasoning_effort ("low"/"medium"/"high"), thinking, temperature, top_p, tools, tool_choice, stop, seed, max_tokens, etc. all pass through to whichever provider serves the model. The proxy sets no default effort; if you don't send the field, the upstream's own default applies. Effort is honoured only where the target model supports it (crazyrouter reasoning models and Claude on wrappy); local LM Studio ignores it.

curl https://llm.hostbun.cc/v1/chat/completions -H "Content-Type: application/json" \
  -d '{"model":"claude-opus-4-8","reasoning_effort":"high",
       "messages":[{"role":"user","content":"prove sqrt(2) is irrational"}]}'

Local models — select model: local or obliterated

Local requests route to llama.cpp on the pbox box (RTX 4090) via pbox.llm.hostbun.cc. Private (data stays on-prem), free. Local model:

  • local (default) → gemma-4-e4b-it-obliterated — Gemma 4 E4B abliterated/uncensored, ~8 GB, multimodal (vision) + tool calling, 32K ctx. The default-loaded model; vision works (send image_url parts incl. base64).
  • gemma → google/gemma-4-26b-a4b — 26B-A4B MoE, vision + tool calling, ~8K ctx.
  • obliterated → qwen3.6-27b-obliterated — 27B abliterated/uncensored, text-only reasoning model. Use a generous max_tokens (e.g. ≥1024) — it spends tokens on a reasoning chain before the answer. 🔒 Gated: send Authorization: Bearer <token> or it returns 401.

Drop it into any OpenAI-compatible client (opencode, Cline, Continue, the OpenAI SDK) as a custom provider — base URL https://llm.hostbun.cc/v1, API key = the obliterated token, model obliterated. /v1/models is open so the model list populates without a key; only obliterated completions need the token.

# opencode — opencode.json (or ~/.config/opencode/opencode.json)
{
  "provider": {
    "bofrid": {
      "npm": "@ai-sdk/openai-compatible",
      "options": { "baseURL": "https://llm.hostbun.cc/v1", "apiKey": "<token>" },
      "models": { "obliterated": { "name": "Qwen3.6-27B Obliterated" } }
    }
  }
}

The local models can't all fit in the 24 GB GPU (gemma and obliterated are ~19 GB each), so LM Studio JIT-swaps: the first request for a model loads it and unloads the previous one. The default local (E4B, ~8 GB) is cheap to swap back in (~5 s); switching to/from a 19 GB model is a ~20–40 s cold start. The legacy explicit path /local/v1/* (and Anthropic /local/v1/messages, native /local/api/v1/chat) still works too.

# either of these reaches the local GPU:
curl https://llm.hostbun.cc/v1/chat/completions -H "Content-Type: application/json" \
  -d '{"model":"local","messages":[{"role":"user","content":"describe"}]}'
curl https://llm.hostbun.cc/local/v1/chat/completions -H "Content-Type: application/json" \
  -d '{"model":"google/gemma-4-26b-a4b","messages":[{"role":"user","content":"describe"}]}'
